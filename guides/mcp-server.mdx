## What it does

The Guard MCP Server intercepts every Claude Code tool call before OS execution.

Supported tools:
- `bash`
- `read_file`
- `write_file`
- `web_fetch`

Each call is routed through the canonical Sector8 Admission Controller. The server either returns `ALLOW` and executes the tool, or returns `DENY` and blocks execution before the tool runs.

There is no separate MCP-only policy path. The MCP server uses the same guardrail engine, invariants, evidence hashing, and telemetry model as the rest of the Guard Module.

## Security guarantees

### Fail-closed on guard exception

If any unhandled exception occurs anywhere in the evaluation path, the Guard MCP Server denies the call.

Returned denial:
- `reason_code: GUARD_INTERNAL_ERROR`
- non-empty `evidence_hash`
- `has_forensic_payload: true`

The tool implementation is never reached on this path.

### Forensic completeness on deny

Every denied MCP response includes:
- `decision`
- `reason_code`
- `evidence_hash`
- `has_forensic_payload`
- `decision_trace_id`

This is enforced by CI through the suite-wide deny invariant.

### Authorization is enforced, not advisory

MCP calls are not approved on intent alone.

The admission path enforces:
- tool registration
- capability checks
- object-level authorization (BOLA)
- function-level authorization (BFLA)
- semantic threat scoring
- segmented-input safety
- request-risk checks including SSRF and sensitive path access
- admission rate limiting

## Installation

```bash
sector8-guard install
```

This writes `.claude/settings.json` for the current project:

```json
{
  "mcpServers": {
    "sector8-guard": {
      "command": "sector8-guard-mcp",
      "args": ["--policy", "enterprise-v1"],
      "env": {
        "SECTOR8_API_KEY": "${SECTOR8_API_KEY}",
        "SECTOR8_CLIENT_ID": "${SECTOR8_CLIENT_ID}"
      }
    }
  }
}
```

## How it works

```
Claude Code tool call
        │
        ▼
Guard MCP Server
        │
        ▼
Admission Controller
  ├─ pre-filter
  ├─ multi-turn / rate checks
  ├─ registration / capability
  ├─ BOLA / BFLA
  ├─ semantic intent
  ├─ segmented-input safety
  └─ request-risk checks
        │
   ┌────┴────┐
   ▼         ▼
 DENY      ALLOW
   │         │
   │         └─ tool executes
   └─ tool never executes
```

## Governed tools

| Tool | What gets checked |
|---|---|
| `bash` | shell injection, destructive commands, obfuscated command payloads, rate limiting |
| `read_file` | sensitive paths, object ownership, authorization |
| `write_file` | content safety, authorization, object scoping |
| `web_fetch` | SSRF, private IP ranges, localhost, DNS resolution to private targets, allowlisted external hosts |

## `web_fetch` protections

`web_fetch` is not guarded by a simple hostname blocklist.

The current request-risk path blocks:
- raw private IP targets
- loopback and localhost
- link-local targets such as `169.254.169.254`
- DNS results that resolve to private IP space
- unsupported URL schemes
- URL credentials
- non-allowlisted external domains

This is designed to stop common SSRF paths, including cloud metadata access.

## Verify the connection

Inside Claude Code:

```
/mcp
```

You should see `sector8-guard` as a connected server.

Then test one allow and one deny:

```
Use the sector8-guard MCP server to run bash: echo hello
Use the sector8-guard MCP server to run bash: rm -rf /
```

Expected:
- `echo hello` → `ALLOW`
- `rm -rf /` → `DENY`

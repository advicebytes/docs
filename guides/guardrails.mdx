## How it works

Every tool call passes through a single gate — the Admission Controller — before reaching the OS. There is no path around it.

```
LLM Output
    │
    ▼
Spotlighting              ← deterministic trust tagging
    │
    ▼
Pre-Filter Classifier     ← fast path: <5ms, BLOCK / PASS / REVIEW
    │
    ▼
Admission Controller      ← 5-check policy gate
    │
    ├─ DENY ──▶ Denial trace emitted
    │           evidence_hash + policy_version + decision_trace_id
    │           has_forensic_payload: true
    │
    └─ ALLOW ─▶ Tool executes
                Full audit telemetry recorded
```

## The Seven Invariants

Non-negotiable system properties. Every deployment enforces all seven.

<AccordionGroup>
  <Accordion title="INV-1 — Single Execution Gate">
    No direct path from LLM output to tool execution. The Admission Controller is the only dispatch path. Enforced at runtime and by a static AST gate in CI.
  </Accordion>
  <Accordion title="INV-2 — Dual Gate Requirement">
    Policy decides intent. Authorization decides capability. Execution requires AND of both — not OR.
  </Accordion>
  <Accordion title="INV-3 — Per-Step Enforcement">
    The Admission Controller runs per tool call, per agent step, after LLM planning, before execution. Cannot be bypassed by chain recursion.
  </Accordion>
  <Accordion title="INV-4 — Untrusted Executor Model">
    The LLM produces candidates. The control plane decides outcomes. LLM confidence is not a security property.
  </Accordion>
  <Accordion title="INV-5 — Spotlighting Before Policy">
    Content is trust-tagged before policy evaluation. Untrusted content cannot promote its own trust level.
  </Accordion>
  <Accordion title="INV-6 — Boolean AND Execution Condition">
    `execute = (policy == ALLOW) AND (admission == PASS)`. A partial pass is a block.
  </Accordion>
  <Accordion title="INV-7 — Trust Inversion Forbidden">
    No component consuming untrusted outputs may be the final authority to permit privileged execution.
  </Accordion>
</AccordionGroup>

## Admission Controller — 5 Checks

Each tool call passes five sequential checks. Failure at any check returns DENY immediately.

| Check | What it evaluates |
|-------|-------------------|
| 1. Policy evaluation | PII, prompt injection, shell injection, data exfiltration |
| 2. Lockdown Mode | 3-tier capability gate (FULL / PARTIAL / OFF) |
| 3. Authorization | RBAC, BOLA (object-level authorization), BFLA (function-level authorization), and tool allowlist enforcement |
| 4. Tool allowlist | Only registered, signed tools may execute |
| 5. Semantic intent | Embedding-based similarity to known threat clusters |

## Pre-Filter

Before the full Admission Controller runs, a fast-path classifier routes requests in under 5ms:

| Decision | Meaning | What happens |
|----------|---------|-------------|
| `BLOCK` | High-confidence malicious | Deny immediately — AC skipped |
| `PASS` | High-confidence benign | Allow immediately — AC skipped |
| `REVIEW` | Ambiguous | Full AC evaluation |

## Spotlighting

All content entering the policy evaluation pipeline is tagged with a trust source before evaluation:

| Source tag | Applied to |
|------------|-----------|
| `[SYSTEM: TRUSTED]` | System prompt content |
| `[USER: CONTEXT]` | Direct user input |
| `[RAG: UNTRUSTED]` | Retrieved document chunks |
| `[TOOL_OUTPUT: UNTRUSTED]` | Results from previous tool calls |
| `[FILE_UPLOAD: UNTRUSTED]` | Uploaded file content |

Untrusted content cannot promote its own trust level.

## What gets blocked

<CardGroup cols={2}>
  <Card title="Prompt Injection" icon="syringe">
    Instructions embedded in user input, RAG documents, or tool outputs designed to redirect LLM behaviour.
  </Card>
  <Card title="Shell Injection" icon="terminal">
    Tool call arguments containing destructive OS commands — `rm -rf`, subprocess calls, process spawning.
  </Card>
  <Card title="PII / PHI Exfiltration" icon="user-shield">
    Requests to transmit personal data, patient records, financial information, or credentials externally.
  </Card>
  <Card title="Data Exfiltration" icon="arrow-up-from-bracket">
    Bulk export, external POST, webhook delivery of internal data to unlisted domains.
  </Card>
</CardGroup>

## Forensic telemetry

Every blocked call produces a tamper-evident record. All five fields are present on 100% of denied records.

This includes fail-closed guard exceptions — if evaluation itself throws, the denial is issued with `reason_code: GUARD_INTERNAL_ERROR` and `has_forensic_payload: true`. The tool is never reached.

```json
{
  "status": "blocked",
  "threat_type": "data_exfiltration",
  "severity": "critical",
  "evidence_hash": "bdbde1558fb883573b0f6e8630ccba1edaf6bed8cb035fa6a48912809149f287",
  "has_forensic_payload": true,
  "policy_version": "eea811e4f6d3c22074d57b3f9560ad13582e15844a01cd0865481ef7e0b38c56",
  "decision_trace_id": "17e797c0-eb5c-4f97-b4a1-2d3e4f5a6b7c"
}
```

| Field | Description |
|-------|-------------|
| `evidence_hash` | SHA-256 of the blocked payload. Tamper-evident. |
| `has_forensic_payload` | `true` on 100% of blocked records. |
| `policy_version` | HMAC-SHA256 signed bundle hash. Proves which policy made the decision. |
| `decision_trace_id` | UUID linking the denial to the full admission trace. |

## Compliance coverage

<Info>
Pre-execution compliance is preventive, not detective. Data never moves before a policy decision is made.
</Info>

| Framework | Controls enforced at tool boundary |
|-----------|-----------------------------------|
| **GDPR** | PII detection, data minimisation, Art. 5/17 audit trail |
| **HIPAA** | PHI patterns, minimum-necessary access, §164.312 logging |
| **PCI-DSS v4** | PAN/CVV detection, CDE path blocking, Req. 10.2 + 6.3.2 |
| **ISO 27001** | A.8.2, A.9.4, A.12.4, A.14.2 |
| **SOC 2 Type II** | CC6.1, CC6.6, CC7.2, CC9.2 |
